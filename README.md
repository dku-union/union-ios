# 유니비 (Union) - 대학생 전용 미니앱 슈퍼앱 플랫폼

> 단국대학교 캡스톤디자인 프로젝트

---

## 1. 기획안

### 1.1 프로젝트 배경

AI와 바이브 코딩의 발전으로 누구나 쉽게 소프트웨어를 만들어 비즈니스 모델을 구현할 수 있는 시대가 되었다. 그러나 **소프트웨어를 만드는 것**과 **시장에 내놓는 것**은 완전히 다른 문제이다.

**시장 진입의 장벽:**
- 앱 스토어 개발자 등록 비용 (Apple 연 $99, Google $25)
- 앱 심사 프로세스의 복잡성과 리젝 리스크
- SEO 설정 및 웹 배포 인프라 구축
- 마케팅 비용 부담 및 효과적인 소재 선정의 어려움
- 초기 사용자 확보의 난이도

이 문제를 해결하기 위해 슈퍼앱 형태의 플랫폼을 기획하였으나, 이미 토스(Toss)라는 거대 기업이 범용 슈퍼앱 시장을 점유하고 있다. 따라서 **대학생이라는 특정 타겟**으로 시장을 좁혀, 대학생 커뮤니티 내에서 빠르게 시장을 선점하는 전략을 택하였다.

### 1.2 프로젝트 개요

**유니비(Union)**는 대학생 전용 미니앱을 담은 슈퍼앱 플랫폼이다. 퍼블리셔(개발자)가 React 기반의 미니앱을 제작하여 플랫폼에 배포하면, 대학생 사용자들이 하나의 앱 안에서 다양한 미니앱을 이용할 수 있다.

**핵심 가치:**
- 퍼블리셔: 앱 스토어 등록 없이 미니앱을 배포하여 빠른 시장 검증 및 MVP 검증 가능
- 사용자: 대학 생활에 필요한 다양한 서비스를 하나의 앱에서 이용

### 1.3 타겟 사용자

| 구분 | 대상 | 설명 |
|------|------|------|
| **퍼블리셔** | 개인 개발자, 학생회, 동아리 등 | 미니앱을 개발하고 플랫폼에 배포하는 주체 |
| **사용자** | 대학생 | 슈퍼앱을 통해 미니앱을 이용하는 최종 사용자 |

### 1.4 대학생 타겟의 이점

- **트렌드 민감성**: 대학생은 새로운 서비스 수용에 적극적
- **학생회 연결**: 축제 앱, 야식행사 앱, 주점 웨이팅 앱, 소개팅 앱 등을 학생회에서 직접 제작하고 배포 가능
- **공동체 기반 확산**: 캠퍼스라는 폐쇄적 공동체 내에서의 시장 진입이 비교적 용이
- **도메인 이해도**: 팀원 전원이 대학생으로서 대학생의 문제를 직접 파악 가능
- **초기 콘텐츠 확보**: 팀이 직접 몇 가지 미니앱을 출시하여 초기 콘텐츠 확보

### 1.5 시장 전략 (SOM/SAM/TAM)

```
TAM (전체 시장): 전국 대학교 대학생 약 190만명
 └─ SAM (유효 시장): 수도권 및 주변 대학교 대학생 약 80만명
     └─ SOM (초기 시장): 단국대학교 재학생 약 2만명
```

**확장 로드맵:** 단국대학교 → 주변 대학교 → 전국 대학교

### 1.6 경쟁사 및 유사 서비스 분석

| 비교 항목 | WeChat 미니프로그램 | 카카오 미니앱 | 토스 미니 | LINE Mini App | **유니비** |
|-----------|---------------------|---------------|-----------|---------------|------------|
| 타겟 | 일반 대중 | 일반 대중 | 금융 중심 | 일반 대중 | **대학생 특화** |
| 미니앱 기술 | WXML/WXSS (독자) | Web | Web | Web (LIFF) | **React (표준)** |
| 초기 시장 진입 | 매우 어려움 | 어려움 | 초대제 | 어려움 | **용이 (학생회)** |
| 개발자 등록 비용 | 무료 | 무료 | 비공개 | 무료 | **무료** |
| 심사 프로세스 | 엄격 | 엄격 | 엄격 | 보통 | **보통 (커뮤니티)** |
| 커뮤니티 특성 | 글로벌 | 국내 전체 | 국내 전체 | 일본/아시아 | **캠퍼스 밀착** |
| 주요 사용자 동기 | 생활 편의 | 생활 편의 | 금융 서비스 | 생활 편의 | **대학 생활 문제 해결** |

**유니비의 차별점:**
1. **캠퍼스 밀착형 플랫폼**: 대학생 커뮤니티 내에서만 동작하는 폐쇄형 생태계로, 학생회/동아리와의 직접 연결이 가능
2. **표준 기술 기반**: WeChat처럼 독자 프레임워크가 아닌 React 표준 기술을 사용하여 퍼블리셔 진입장벽 최소화
3. **학교 인증 기반 신뢰**: 대학교 메일 인증을 통한 사용자 신원 보장으로 높은 커뮤니티 신뢰도
4. **낮은 진입장벽**: 학생회/동아리가 코드 작성 없이도 바이브 코딩을 활용해 미니앱 제작 가능

### 1.7 비즈니스 모델

```
┌─────────────────────────────────────────────────────────┐
│                    수익 구조 (Revenue Model)               │
├──────────────┬──────────────┬───────────────────────────┤
│   Phase 1    │   Phase 2    │         Phase 3           │
│  (출시~6개월) │ (6개월~1년)   │        (1년 이후)          │
├──────────────┼──────────────┼───────────────────────────┤
│ • 무료 운영   │ • 프리미엄    │ • 앱 내 결제 수수료       │
│ • 사용자 확보 │   퍼블리셔    │   (미니앱 내 결제의 5~15%) │
│   에 집중     │   구독 모델   │ • 광고 수익               │
│              │ • 대학 제휴   │   (홈 배너, 추천 앱 광고)  │
│              │   (B2B)      │ • 데이터 인사이트 판매     │
│              │              │   (익명화된 트렌드 분석)    │
└──────────────┴──────────────┴───────────────────────────┘
```

| 수익원 | 설명 | 예상 시점 |
|--------|------|-----------|
| **프리미엄 퍼블리셔 구독** | 고급 애널리틱스, 우선 심사, A/B 테스트 등 | Phase 2 |
| **대학 제휴 (B2B)** | 학교 공식 서비스 채널로 활용 (학사일정, 공지 등) | Phase 2 |
| **앱 내 결제 수수료** | 미니앱 내 결제 발생 시 플랫폼 수수료 부과 | Phase 3 |
| **광고 수익** | 홈 화면 배너, 추천 앱 광고 슬롯 판매 | Phase 3 |
| **트렌드 인사이트** | 대학생 소비/관심 트렌드 데이터 (익명화) 판매 | Phase 3 |

### 1.8 기술 스택

| 영역 | 기술 | 선정 사유 |
|------|------|-----------|
| 퍼블리셔 대시보드 | Next.js | SSR 지원, React 생태계 공유 |
| 사용자 앱 (iOS) | Swift | 네이티브 성능, WebView 제어 |
| 사용자 앱 (Android) | Kotlin | 네이티브 성능, WebView 제어 |
| 미니앱 런타임 | React → 패키징 | 퍼블리셔 진입장벽 최소화 |
| 미니앱 배포 | 클라우드 스토리지 + CDN | 빠른 로딩, 글로벌 엣지 캐싱 |
| 인증 | 자체 OAuth 2.0, mTLS | 미니앱-슈퍼앱 간 보안 통신 |
| 백엔드 API | (미정) | - |
| 데이터베이스 | (미정) | - |
| 캐시/세션 | (미정) | - |
| 푸시 알림 | APNs (iOS) / FCM (Android) | 네이티브 푸시 지원 |

## 4. 미니앱 SDK (Union Bridge API)

### 4.1 SDK 개요

미니앱(WebView)이 슈퍼앱의 네이티브 기능에 접근하기 위한 JavaScript Bridge SDK이다. `window.postMessage` 기반으로 WebView와 네이티브 앱 간 양방향 통신을 수행한다.

```
미니앱 (JS) ──→ Union SDK ──→ postMessage ──→ Native Bridge Handler ──→ 네이티브 기능
                                                      │
미니앱 (JS) ←── Callback ←── postMessage ←────────────┘
```

### 4.3 권한 모델 (Permission Scopes)

미니앱이 접근할 수 있는 사용자 데이터 및 디바이스 기능은 권한 범위(Scope)에 의해 제어된다.

| Scope | 설명 | 동의 필요 | 민감도 |
|-------|------|-----------|--------|
| `user.profile` | 기본 프로필 (닉네임, 프로필 이미지) | 자동 (기본 제공) | 낮음 |
| `user.email` | 이메일 주소 | 명시적 동의 | 중간 |
| `user.university` | 학교 정보 (학교명, 학과) | 명시적 동의 | 중간 |
| `device.location` | 위치 정보 | 명시적 동의 | 높음 |
| `device.camera` | 카메라 접근 (QR 스캔) | 명시적 동의 | 높음 |
| `device.storage` | 로컬 저장소 | 자동 (앱별 격리) | 낮음 |

**권한 부여 흐름:**
```
미니앱 실행 → 필요 권한 확인 → 미동의 권한 존재?
                                  │
                    ┌─── Yes ─────┘
                    ▼
              동의 팝업 표시
              "XX앱이 다음 정보에
               접근하려 합니다:
               • 학교 정보
               • 위치 정보"
              ┌──────┬──────┐
              │ 허용  │ 거부  │
              └──┬───┘──┬───┘
                 │      │
              권한 저장  미니앱에
              미니앱     제한된 기능
              정상 실행  으로 실행
```

### 4.4 미니앱 업데이트 전략

| 전략 | 설명 | 적용 조건 |
|------|------|-----------|
| **자동 업데이트** | CDN 기반이므로 사용자가 미니앱 실행 시 항상 최신 버전 로드 | 기본 동작 |
| **점진적 롤아웃** | 새 버전을 전체 사용자의 10% → 50% → 100%에게 순차 배포 | 퍼블리셔 선택 |
| **강제 업데이트** | 최소 지원 버전 미만일 경우 업데이트 강제 | 보안 패치 등 |

### B. 용어 사전

| 용어 | 정의 |
|------|------|
| 슈퍼앱 (Super App) | 다양한 서비스를 하나의 앱에서 제공하는 플랫폼 앱 |
| 미니앱 (Mini App) | 슈퍼앱 내에서 실행되는 경량 웹 애플리케이션 |
| 퍼블리셔 (Publisher) | 미니앱을 개발하고 플랫폼에 배포하는 주체 |
| Union SDK | 미니앱이 슈퍼앱 네이티브 기능에 접근하기 위한 JavaScript Bridge SDK |
| mTLS | 상호 TLS (Mutual TLS), 클라이언트와 서버가 서로의 인증서를 검증하는 보안 프로토콜 |
| OAuth 2.0 | 사용자 인증 및 권한 부여를 위한 표준 프로토콜 |
| CDN | Content Delivery Network, 전 세계에 분산된 서버 네트워크를 통한 콘텐츠 전달 |
| 롤아웃 (Rollout) | 새 버전을 전체 사용자에게 점진적으로 배포하는 전략 |
| 샌드박싱 (Sandboxing) | 미니앱 간 데이터를 격리하여 보안을 보장하는 기술 |
| CSP | Content Security Policy, 웹 리소스의 로딩 정책을 제어하는 보안 메커니즘 |
| Cold Start | 추천 시스템에서 신규 사용자/아이템에 대한 데이터 부족 문제 |
